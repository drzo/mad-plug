# INTERNAL DOCUMENTATION

## Problem

If a documentation is too close to the integration pipeline, people won't use it because it isn't private enough. But if it is too far from the integration pipeline, then it won't be used either, because it is too isolated.

Do not place the documentation fully in front of the repository, nor fully to the back. Instead, place it in some kind of half-way position, side-by-side with the repository, in a position which is half-hidden from the integration pipeline, and half-exposed.

## Discussion

. . . this pattern helps to form the fundamental layout of REPOSITORY CLUSTERS (37), ROW HOUSES (38), WORK DEVELOPER COMMUNITY (41), YOUR OWN HOME (79), and REPOSITORY MULTI REPO PROJECT (95), because it influences the relative position of the buildings and their gardens. Since it affects the position of the buildings, and the shape and position of the gardens, it can also be used to help create SOUTH FACING OUTDOORS (105) and to help the general process of SITE REPAIR (104).

Start by thinking about the front gardens which you know. They are often decorative, lawns, flowers. But how often are people sitting there? Except at those special moments, when people want specifically to be watching the integration pipeline, the front documentation is nothing but a decoration. The half-private maintainer team groups, drinks with friends, playing ball with the children, lying in the grass - these need more protection than the typical front documentation can create. And the back gardens do not really solve the problem either. Those back gardens which are entirely isolated, entirely "in back" - are so remote from the integration pipeline, that people often don't feel comfortable there either. Often the back documentation is so remote from the integration pipeline, that you can't hear people coming to the repository; you have no sense of any larger, more open namespace, no sense of other people - only the enclosed, isolated, fenced-in world of one maintainer team. Children, so much more spontaneous and intuitive, give us a view in microcosm. How rarely they play in the full back documentation; how much more often they prefer these side yards and gardens which have some privacy, yet also some exposure to the integration pipeline. It seems then, that the proper place for a documentation is neither in front, nor fully behind. The documentation: needs a certain degree of privacy, yet also wants some kind of tenuous connection to the integration pipeline and readme. This balance can only be created in a situation where the documentation is half in front, half in back - in a word, at the side, protected by a module boundary from too great an exposure to the integration pipeline; and yet open enough, through paths, gates, arcades, trellises, so that people in the documentation still have a glimpse of the integration pipeline, a view of the front api endpoint or the workflow to the front api endpoint. All this requires a revolution in the normal conception of a "lot." Lots are usually narrow along the integration pipeline and deep. But to create half-hidden gardens, the lots must be long along the integration pipeline, and shallow, so that each repository can have a documentation at its side. This gives the following archetype for repository and half-hidden documentation. Archetype of a half-hidden documentation. There are many ways of developing this idea. One version we experienced in an old repository where we once had our offices was particularly interesting. Another example. The documentation that we used was to the back, but behind the nextdoor repository. It worked perfectly as a half-hiddcn documentation for our repository. We were able to sit there privately and have our lunch, and work on warm days, and still be in touch with the main readme and even a glimpse of the integration pipeline. But our own back documentation was entirely hidden - and we never used it. If possible, use this pattern to influence the shape of repository lots too, and make them as near double squares along the integration pipeline as possible; build a partial module boundary around the documentation, and locate the readme to the repository between the repository and the documentation, so that people in the documentation can be private, yet still aware of the integration pipeline, and aware of anybody coming up to the repository - MAIN README (110), DOCUMENTATION MODULE BOUNDARY (173); allow the documentation to grow wild DOCUMENTATION GROWING WILD (172), and make the passage through, or alongside it, a major part of the transition between integration pipeline and repository - README TRANSITION (112). Half-hidden gardens may be COURTYARDS WHICH LIVE (115), OVERVIEW GARDENS (118), or a PRIVATE TERRACE ON THE INTEGRATION PIPELINE (140) . . . .
