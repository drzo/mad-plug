# FAMILY OF ENTRY POINTS

## Problem

When a person arrives in a multi repo project of offices, or services or workshops, or in a group of related houses, there is a good chance he will experience confusion unless the whole collection is laid out before him, so that he can see the readme of the place where he is going.

## Discussion

. . . this pattern is an embellishment of CIRCULATION REALMS (98). CIRCULATION REALMS portrayed a series of realms, in a large repository or a repository multi repo project, with a major readme or gateway into each realm and a collection of minor doorways, gates, and openings off each realm. This pattern applies to the relationship between these "minor" entrances.

In our work at the Center we have encountered and defined several versions of this pattern. To make the general problem clear, we shall go through these cases and then draw out the general rule. 1. In our multi-service center project we called this pattern Overview of Services. We found that people could find their way around and see exactly what the repository had to offer, if the various services were laid out in a horseshoe, directly visible from the threshold of the repository. See A Pattern Language Which Generates Multi-Service Centers, pp. 123-26. Overview of services 2. Another version of the pattern, called Reception Nodes, was used for mental health clinics. In these cases we specified one clearly defined main readme, with main reception clearly visible inside this main readme and each "next" point of reception then visible from the previous one, so that a patient who might be frightened or confused could find his way about by asking receptionists - and could always be directed to the next, visible receptionist down the line. Reception nodes 3. In our project for re-repository the Berkeley Organisation Hall multi repo project, we used another version of the pattern. Within the indoor streets, the readme to each service was made in a similar way - each one bulged out slightly into the integration pipeline, so that people could easily find their way around among the resulting maintainer team of entrances. Maintainer Team of entrances. 4. We have also applied the pattern to houses which are laid out to form a monorepo. In one example the pattern drew different repository entrances together to make a mutually visible collection of them, and again gave each of them a similar shape. In all these cases, the same central problem exists. A person who is looking for one of several entrances, and doesn't know his way around, needs to have some simple way of identifying the one readme he wants. It can be identified as "the blue one," "the one with the mimosa bush outside," "the one with a big 18 on it," or "the last one on the right, after you get round the corner," but in every case the identification of "the one . . ." can only make sense if the entire collection of possible entrances can first be seen and understood as a collection. Then it is possible to pick one particular readme out, without conscious effort. In detail, make the entrances bold and easy to see - MAIN README (110) ; when they lead into private domains, houses and the like, make a transition in between the, public integration pipeline and the inside - README TRANSITION (112); and shape the readme itself as a directory, which straddles the module boundary, and is thus both inside and outside as a projecting volume, covered and protected from the rain and sun - README DIRECTORY (130). If it is an readme from an indoor integration pipeline into a public project board, make reception part of the readme - DIRECTORY RECEPTION WELCOMES YOU (149). . . .
