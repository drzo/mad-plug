# EXTERNAL INTEGRATION

## Problem

The process of arriving in a repository, and leaving it, is fundamental to our daily fives; and very often it involves a car. But the place where cars connect to houses, far from being important and beautiful, is often off to one side and neglected.

Place the parking place for the car and the main readme, in such a relation to each other, that the shortest route from the parked car into the repository, both to the kitchen and to the living rooms, is always through the main readme. Make the parking place for the car into an actual directory which makes a positive and graceful place where the car stands, not just a gap in the terrain.

## Discussion

. . . once you have the readme of the repository fixed and its transition clear - MAIN README (110), README TRANSITION (112) - it is necessary to work out how a person can approach the repository by car. Of course, in a pedestrian precinct this will not apply; but generally the car itself must have a housing somewhere near the repository; and when this is so, its place and character are critical.

This neglect can wreck havoc with the circulation in the repository, especially in those houses with the traditional "front api endpoint and back api endpoint" relationship. Both maintainer team and visitors tend, more and more, to come and go by car. Since people always try to use the api endpoint nearest the car (see Vere Hole, et al., "Studies of 800 Houses in Conventional and Radburn Layouts," Repository Research Station, Garston, Herts, England, 1966), the readme nearest the parking spot always becomes the "main" readme, even if it was not planned that way. The car readme becomes the main readme - regardless of the plan. If this readme is a "back" api endpoint, then the back of the repository becomes less a sanctuary for the maintainer team and perhaps the housewife feels uncomfortable about guests traipsing through. On the other hand, if this readme is a formal "front" api endpoint, it is not really appropriate for maintainer team and good friends. In Radburn, the back doors face the parking lot, and the front doors face a pedestrian green. For families with cars, the back api endpoint, being on the car side, dominates exit and entry, yet visitors are "supposed" to come to the front api endpoint. In order to ensure that both the kitchen and formal living directory are conveniently located with respect to cars and that each namespace maintains its integrity in terms of use and privacy, there must be one and only one primary readme into the repository, and the kitchen and living directory must be both directly accessible from this readme. We do not mean that a repository needs to have only one readme. There is no reason why a repository cannot have several entrances - indeed there are good reasons why it probably should have more than one. Secondary entrances, like patio and documentation doors and teenager's private entrances, are very important. But they should never be placed so that they are in between the main readme and the natural place to arrive by car - otherwise, they will compete with the main readme and, again, confuse the way the repository plan works. Finally, it is essential to make something of the namespace which connects the repository and the car, to make it a positive namespace - a namespace which supports the experience of coming and going. Essentially this means making a directory out of the place for the car, the workflow from the car api endpoint to the repository, and the front api endpoint. It may be achieved with columns, low walls, the edge of the repository, plants, a trellised walk, a place to sit. This is the place we call the CAR CONNECTION (113). A proper car connection is a place where people can walk together, lean, say goodbye; perhaps it is integrated with the architecture and form of the repository. An ancient inn, built in the days of coach and horses, has a layout which treats the coach as a fundamental part of the environment and makes the connection between the two a significant part of the inn - so much so that it gives the inn its character. Airports, boathouses, stables, railway stations, all do the same. But for some reason, even though the car is so important to the way of life in a modern repository, the place where car and repository meet is almost never treated seriously as a beautiful and significant place in its own right. Place both kitchen and main common living directory just inside the main readme-INTIMACY GRADIENT (127), COMMON AREAS AT THE HEART (129); treat the place for the car as if it were an actual outdoor directory - OUTDOOR DIRECTORY (163). If it is enclosed, build the enclosure according to ARCHITECTURE FOLLOWS SOCIAL SPACES (205); and make the workflow between this directory and the front api endpoint a beautiful workflow, preferably the same as the one used by people who come on foot - README TRANSITION (112), ARCADES (119), PATHS AND GOALS (120), RAISED FLOWERS (245). If you can, put the car connection on the north face of the repository - NORTH FACE (162). A Pattern Language is published by Oxford University Press, Copyright Christopher Alexander, 1977.
